# IPO Reflection — 의사결정 복기

이 하네스가 한 번의 분석 도구를 넘어 **학습하는 시스템**이 되는 메커니즘이다. 판정 당시의 논거·확신도·크럭스·진입 계획과 상장 후 실제 전개를 대조해, 다음 판정을 바꿀 교훈만 추출한다. (2차 시장 하네스 trading-agent의 `trade-reflection`과 같은 골격 — IPO 변형.)

## 핵심 원칙: 결과론 금지

좋은 결정이 나쁜 결과를 낳을 수 있고, 나쁜 결정이 운 좋게 수익을 낼 수 있다. 평가 대상은 **결과가 아니라 당시 정보 기준의 의사결정 품질**이다:

- 따상이 났지만 "pop +30% 추격 금지" 규율을 어기고 산 것 → **나쁜 결정** (다음엔 그 운이 없다)
- 손실이 났지만 가설훼손 신호 발동 즉시 추가 진입을 중단했다 → **좋은 결정** (시스템이 작동했다)
- "따상인데 왜 1차를 생략했나"는 결과론이다 — 당시 게이트·판결 기준으로 생략이 옳았다면 좋은 결정이다.

## 수익률 기준선 — IPO는 두 개다

IPO 성과는 "어디서 샀는가"에 따라 다른 게임이다. 둘 다 산출한다:

1. **공모가 대비:** 청약 배정을 받았을 때의 성과. 판결(공모가 대비 가치)의 적중도를 재는 기준.
2. **기준 진입가 대비:** 계획이 실제 진입을 상정한 가격(1차 구간 실행 시 상장일 시초가/초기 거래가, 1차 생략 시 2차 구간 가격대). 시장가 매수가 현실인 투자자(미국 IPO의 한국 투자자 등)의 실제 성과를 재는 기준.

**벤치마크 알파 필수:** 같은 기간 벤치마크(미국 IPO면 S&P500/나스닥, 한국이면 KOSPI/KOSDAQ) 등락을 대조해 알파를 산출한다. 절대 수익률만으로 평가하지 않는다 — 강세장의 +20%는 실력이 아닐 수 있고, 약세장의 -5%는 잘 방어한 것일 수 있다.

## 복기 절차

1. `decisions/journal.md`에서 대상 판정을 찾는다 (사용자가 특정 안 하면 `결과:` 미기록 항목을 보여주고 확인). 상장 전(거래 개시 전)이면 복기 불가 — 그 사실만 알리고 종료한다.
2. 상장 후 가격 전개를 수집한다: 현재가·상장일 시초가/종가·기간 고점/저점을 WebSearch(미국)·네이버 증권/KRX(한국, 막히면 `insane-search`)로 확인하고, 같은 기간 벤치마크 레벨을 함께 수집해 알파를 계산한다. 출처·시점을 명시한다.
3. **4축 대조 (IPO판):**
   | 축 | 질문 |
   |----|------|
   | 논거·크럭스 적중 | 판결의 크럭스가 실제로 어느 방향으로 해소됐는가. 핵심 논거(옵션 프리미엄·락업 오버행·수요예측 신호 등)가 전개됐는가. 틀렸다면 정보 부족이었나 추론 오류였나 |
   | 반증·가설훼손 작동 | 가설훼손 체크포인트가 발동했는가, 발동 시 계획대로 추가 진입 중단/축소했는가(했어야 했는가). 반증 조건이 추적 가능한 형태(지표+임계값+시점)였는가 |
   | 진입 설계 적합 | 실행 게이트 판정(상장일 매수 가능 여부)이 맞았는가. 분할 구간·비중이 실제 가격 경로(pop, 안정조작 종료, 락업/의무보유확약 해제, 지수편입)와 맞았는가. 잠재 손실 추정이 실제 변동성과 맞았는가 |
   | 누락 변수 | 당시 파이프라인이 못 본 것은 무엇인가(수요예측 신호 오독, 지수편입 시점, 매크로, 경쟁 이벤트), 어느 에이전트의 사각지대였는가 |
4. `decisions/journal.md`의 해당 항목 `결과:` 줄을 갱신한다 (이 줄만 수정 — 나머지 기록은 보존). 형식: `결과: [복기 {YYYY-MM-DD}] 공모가 대비 {±X}% · 기준 진입가 대비 {±Y}%, 벤치마크 {±Z}% (알파 {±W}%p) — {의사결정 품질 1줄 평가}`. 파이프라인이 남긴 `[중간점검 ...]` 줄이 있으면 복기 확정으로 덮어쓴다.
5. **캘리브레이션 집계:** 이 결정의 확신도(p)와 결과 채점(o: 적중=1 / 부분=0.5 / 실패=0 — **벤치마크 대비 알파 부호가 판정 방향과 일치**하면 적중, 반대면 실패, 애매하면 부분)을 `decisions/calibration.md`에 1행 추가하고 버킷 요약을 재계산한다. 목적: 확신도가 실제 적중률과 일치하는지(캘리브레이션)를 누적 측정해, 과신/과소신이 확인되면 확신도→비중 매핑을 재보정한다. 파일이 없으면 아래 포맷으로 생성한다(두 하네스 공용 — 트레이딩 쪽도 동일 포맷 사용 예정):

   ```markdown
   # 판정 캘리브레이션 대장 (두 하네스 공용)
   | 날짜 | 하네스 | 종목 | 판정 | 확신도(p) | 결과(o: 1/0.5/0) | Brier (p−o)² |
   ## 버킷 요약 (복기 시 재계산)
   | 확신도 버킷 | 건수 | 평균 확신도 | 실제 적중률 | 과신/과소신 판정 |
   ```
6. 교훈을 추출해 `decisions/lessons.md`에 append한다.
7. **학습 코퍼스 커밋:** `git add decisions/ && git commit -m "복기: {회사} IPO"` — 오케스트레이터 밖에서 이 스킬이 단독 실행될 때도 커밋을 생략하지 않는다(무버전 학습 코퍼스는 단일 실패점).

## 교훈 추출 기준 — 일반화 가능한 것만

- **교훈인 것:** 패턴 — "pop +30% 초과 구간의 1차 추격은 2회 연속 상장 후 4주 고점 매수였다", "의무보유확약 비율이 낮은 IPO의 첫 1개월 변동성을 반복적으로 과소평가했다", "락업 해제 D-2주에 잡은 4차 구간이 반복적으로 조기였다", "free float 갭 미해결인 종목의 확신도 상한을 안 낮췄다"
- **교훈이 아닌 것:** 1회성 불운("상장 직후 거시 쇼크"), 결과론("따상 났으니 1차를 풀로 살 걸"), 종목 고유 사건의 과잉 일반화
- 같은 패턴이 2회 이상 관찰되면 교훈의 확신도를 올린다. 1회 관찰은 "가설" 등급으로 적는다.

## 교훈 포맷 (`decisions/lessons.md`에 append)

실제 `decisions/lessons.md`의 진화된 구조를 그대로 따른다 — `적용 대상`은 제목 헤더에 붙이고, 본문은 무슨 일 / 왜(근본) / 어떻게 적용 / 근거 사례 필드로 쓴다:

```markdown
---
## {YYYY-MM-DD} — {교훈 제목 1줄} | 적용 대상: {대상}
- **등급: [가설(1회 관찰) | 교훈(2회+ 관찰)]**
- **무슨 일:** {복기에서 관찰한 사건 — journal 날짜·종목·판정·확신도 포함}
- **왜(근본):** {근본 원인 — 왜 이 패턴이 반복되는가/문제인가}
- **어떻게 적용:** {다음 판정·계획에서 무엇을 다르게 해야 하는가 — 대상 에이전트별로 행동 가능한 형태}
- 근거 사례: {journal의 날짜·종목·판정}
```

`적용 대상`(대상: `bull/bear(analyst) | data-collector | fact-checker | judge | execution | verdict-reviewer | 전체`)이 중요하다 — 각 에이전트는 시작 시 자기에게 적용되는 교훈만 골라 반영한다. 두 하네스가 모두 해당하면 **전체**로 표기한다.

## 안티패턴

- 교훈 남발: 모든 복기에서 교훈을 만들 필요 없다. "시스템이 설계대로 작동했고 결과는 확률의 영역"이 결론이면 그게 정답이다.
- 기존 교훈과 모순되는 새 교훈이 생기면 둘 다 보존하고 모순을 명시한다 — 다음 복기가 판정한다. 기존 교훈 삭제는 같은 패턴의 반례가 2회 이상일 때만.
- 복기 시점이 너무 이르면(예: 상장 1주 내) "중간점검"으로만 기록하고 교훈 추출은 보류한다 — IPO 초기는 수급·가격발견 노이즈가 지배해 의사결정 품질 신호가 약하다.
